## 3. Milestone Gates

Use `pass`, `fail`, or `partial`. Time is relative to the agent start: `T+00:00`.

| Milestone | Definition | Status | Reached At | Evidence | Verifier Action |
|---|---|---|---|---|---|
| M0 | Run contract fixed: target repository, resolved revision, implementation requirement, dataset/task, official entry or wrapper command, expected output, and success criterion are recorded. |  |  |  | Open the archived run contract or launcher snapshot and confirm all required fields exist. |
| M1 | Docker runtime usable: container starts, Python interpreter, pip, and at least one necessary import, version, device, or environment check completed inside Docker with exit code `0`. |  |  |  | Open the runtime check log and confirm the command exit code is `0`. |
| M2 | Official entry launched inside Docker: the repository-provided run script, main entry point, notebook wrapper, README command, or unit/integration test path was actually executed through an archived Docker wrapper. |  |  |  | Compare the archived command with repository files or documentation and confirm it is an official entry. |
| M3 | Intermediate artifact exists: at least one human-inspectable intermediate artifact exists, such as processed data, generated documents, index, cache, test log, preprocessing output, or partial predictions. |  |  |  | Open the referenced intermediate artifact and confirm it is non-empty and generated by this run. |
| M4 | Output and outcome evidenced: at least one human-inspectable main output exists, and the primary metric is reported; or a terminal blocker is tied to the exact failed command and complete stderr or traceback. |  |  |  | Open the referenced output and metric log, or verify the failed command and error evidence are fully captured. |
| M5 | Archive complete: trajectory or session evidence, launcher log, key scripts, key artifacts, patch or diff, final note, completed checklist, and all I2/I3 errors are archived and classified. |  |  |  | Inspect the archive directory and confirm all required files and I2/I3 error entries exist. |

### Gate Rules

- If M2 is `fail`, final judgment cannot be `success`.
- If M4 is `fail`, final judgment cannot be `success`.
- If M5 is `fail`, final judgment cannot be `success`.

### `partial_success` Triggers

Assign `partial_success` if any of the following apply:

- M2 and M4 are both `pass`, but at least one I2 error remains unrecovered.
- The primary metric is reported, but not all success criterion subitems are met.
- A reference or paper misunderstanding caused implementation drift, while the core flow still runs.
- In a multi-task run, at least one task succeeded and at least one task failed.

### Reference Misread Rule

A reference misread means the agent misunderstood the original paper, report, code comments, benchmark protocol, hyperparameters, data processing flow, or evaluation protocol, causing the implementation to diverge from the intended reproduction. Record this as type `LOGIC`, attribution `AGENT(MISREAD)`, and cite the exact reference location in `Notes`.

### Notes

- M3 must point to a file or directory that a human can directly open or inspect.
- M4 requires either success evidence or blocker evidence. "The run is still running" is not valid M4 evidence.
- Prefer exact paths, filenames, log excerpts, command IDs, and exit codes over narrative evidence.
- The `Verifier Action` field should be a single direct check, not a long procedure.

### Evidence Examples

- M0: archived run contract, launcher snapshot, README section reference, written success criterion.
- M1: Docker build/run log, `python -c` output inside Docker, `python -m pip --version`, successful import check, CUDA or device visibility check.
- M2: exact command for `run.py`, `main.py`, `train.sh`, `evaluate.py`, `pytest`, `tests_*.py`, or notebook wrapper.
- M3: processed dataset shard, tokenized data file, generated corpus, document index, cache database, preprocessing log, unit test log.
- M4: `.ckpt`, `.pt`, `.pth`, `.bin`, `.safetensors`, `.parquet`, `.json`, `.csv`, `events.out.tfevents*`, predictions, result directory, evaluation output, metric table, final score line, or full stderr/traceback for a blocker.
- M5: trajectory or session transcript, `launcher.log`, saved patch or diff, copied run script, final note, completed checklist, output path inventory.

## 4. Error Classification

### 4.1 Fields

**Outcome Code**

- `COMPLETE`: The run reached a valid completed state.
- `IF`: Invalid format, malformed agent output, or unusable structured response.
- `IA`: Invalid action, invalid tool call, or rejected tool usage.
- `TLE`: Explicit time or step budget exceeded.
- `OTHER`: Any other non-complete state.

**Phase**

- `SETUP`: Environment preparation, dependency installation, data download, or preprocessing.
- `EXECUTION`: Main run, training, inference, generation, testing, or evaluation script execution.
- `EVAL`: Result parsing, metric calculation, judgment, or archive review.

**Impact**

- `I0`: Informational; no effect on the run.
- `I1`: Local degradation; does not affect final judgment.
- `I2`: Major degradation; usually leads to `partial_success`.
- `I3`: Blocking failure; prevents the core success criterion.

**Recovery Status**

- `RECOVERED`
- `NOT_RECOVERED`

**Attribution**

- `AGENT(MISREAD)`: The agent misread a paper, algorithm, hyperparameter, data processing flow, or evaluation protocol.
- `AGENT(PLAN)`: The agent decomposed the task incorrectly, used the wrong order, or omitted a necessary step.
- `AGENT(TOOL)`: The agent chose the wrong tool, used invalid parameters, repeated an ineffective tool call, or ignored an available tool.
- `AGENT(LOOP)`: The agent repeated an unproductive pattern without recognizing the failure mode.
- `AGENT(HALLUC)`: The agent invented a nonexistent path, file, API, parameter, or repository behavior and acted on it without verification.
- `AGENT(STOP)`: The agent stopped early while feasible next actions remained.
- `AGENT(OTHER)`: Other agent-caused issue; explain in `Notes`.
- `REPO`: Target repository code, scripts, documentation, packaging, or tests caused the issue.
- `INFRA`: Runtime, OS, hardware, filesystem, scheduler, container, or compute environment caused the issue.
- `PROVIDER`: External model, API, authentication, rate limit, network, or policy issue.
- `EXTERNAL`: Other external dependency issue.
- `MIXED`: Multiple parties contributed.
- `UNCONFIRMED`: Attribution is not yet confirmed.

### 4.2 Error Type Codes

- `ENV`: Interpreter, OS, device, shell, compiler, CUDA, or runtime mismatch.
- `DEP`: Missing or incompatible package, import, binary, or system dependency.
- `PATH`: Wrong working directory, file path, dataset path, checkpoint path, or artifact path.
- `CFG`: Wrong config, CLI argument, schema, environment variable, or credential wiring.
- `PROVIDER`: External service model, API, authentication, rate limit, network, or policy issue.
- `TOOLING`: Tool misuse, rejected output format, agent runtime issue, or wrapper issue.
- `LOGIC`: Repository bug, script bug, parser bug, compatibility bug, or implementation drift from reference material.
- `CONTROL`: Timeout, repeated loop, long stall, drift, or no-progress behavior.
- `DATA`: Missing, corrupted, inaccessible, incompatible, or license-blocked data.

### 4.3 Type Compression Rule

- Prefer one primary error type per row.
- Group authentication, rate limit, network, and external model issues under `PROVIDER`.
- Group timeout, format loops, drift, and stalls under `CONTROL` unless `Notes` needs more detail.
- Group reference or paper misunderstandings under `LOGIC`, with attribution `AGENT(MISREAD)`.

**Failure Location requirement:** identify the precise location where the failure occurred. Prefer one of:

- Script file and line number, such as `train.py:L147`.
- Function or method name, such as `DataLoader.__iter__`.
- Final traceback line.
- Exact command and exit code, such as `pip install torch==2.1.0 -> exit 1`.
- Agent decision step, such as `step 12: agent skipped preprocessing and ran training directly`.
